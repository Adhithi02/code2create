# Audio Identification & Source Detection System



#### Describe your approach here. Keep it short and clear.

- Each WAV file under the genre folders (blues/, classical/, country/, disco/, hiphop/, jazz/, metal/, pop/, reggae/, rock/) is scanned with pathlib and catalogued into a pandas DataFrame keyed by song_id (derived from the genre.NNNNN.wav filename pattern). Three extraction strategies were evaluated: (1) Classical Shazam-style fingerprinting — librosa.stft produces a log-magnitude spectrogram, scipy.signal.find_peaks detects spectral peaks, and combinatorial peak pairs are hashed as (f1, f2, delta_t) tuples; fast, no GPU, deterministic, ~200 hashes per 30s clip. (2) Deep embeddings (VGGish 128-dim or CNN14/PANNS 2048-dim), which encode the full mel-spectrogram into a dense vector robust to noise. (3) Hybrid — both methods run in parallel at index time. The hybrid approach is used in production: hash postings are stored in a SQLite table (song_id, hash_bucket, time_offset) for persistence and mirrored into an in-memory Python dict for O(1) lookups; CNN14 embeddings are persisted as .npy arrays and loaded into a FAISS IndexFlatIP for similarity search. At ~1,000 songs the full postings table is ~200K entries (~50 MB RAM), well within a single process. LMDB and Redis were considered for persistence but rejected at this scale due to added operational complexity with no throughput benefit.

- Queries go through a two-stage pipeline evaluated against three design choices. Option A — hash-only: the query clip's fingerprint hashes are looked up in the postings index and votes are accumulated in a hit_count[song_id][time_shift] histogram; the peak bin declares a match when aligned_hits / total_query_hashes exceeds a threshold. Fast (sub-10ms) but fails below ~5 dB SNR when peak density collapses. Option B — embedding-only: a CNN14 vector is computed for the query and compared against all indexed song vectors via FAISS cosine similarity; robust to noise but ~40–80ms per query on CPU and requires the full embedding pass for every query. Option C (chosen for production) — hybrid two-stage: Stage 1 runs the hash histogram; if the top hit ratio exceeds a calibrated threshold the result is returned immediately without touching the embedding layer. Stage 2 runs only when Stage 1 fails, using FAISS cosine similarity to retrieve the top-k embedding candidates. Final confidence is a weighted blend score = alpha * hash_hit_ratio + (1 - alpha) * embedding_cosine_sim, with alpha tuned on the labeled validation split. Three confidence-scoring strategies were compared: fixed scalar threshold (brittle, requires manual recalibration per noise level), per-genre threshold (better accuracy, high maintenance), and a logistic regression trained on (hit_ratio, cosine_sim) pairs from the test set (production choice — data-driven, automatically adapts, adds under 1ms overhead). Queries below the confidence floor are returned as unknown.

- Scalability is addressed across three axes. Index size: at 1,000 songs the in-memory hash dict and FAISS FlatIP index both fit comfortably in RAM; the scale-up path to 10,000+ songs switches FAISS to IndexIVFFlat (approximate nearest-neighbour with a trained coarse quantizer, ~5–10x faster retrieval at under 1% accuracy loss) or Hnswlib (graph-based ANN, lowest query latency at very large scale). Query throughput: CPU-bound feature extraction (librosa, CNN14 inference) runs in a concurrent.futures.ProcessPoolExecutor with 4–8 workers to bypass the GIL; I/O tasks (WAV reads, DB writes) run in a ThreadPoolExecutor. Three concurrency models were evaluated: threading with RWLock (simple, GIL-limited for CPU work), multiprocessing with ProcessPoolExecutor (true parallelism, chosen), and Celery with Redis as a task broker (decoupled, horizontally scalable, added only when concurrent query load exceeds ~50 req/s). The in-memory hash index is protected by a read-write lock; index rebuilds use a copy-on-write snapshot so live queries are never stalled. The FastAPI server exposes /query and /health endpoints; Flask was considered but rejected in favour of FastAPI's async request handling and automatic OpenAPI documentation.

- Four mechanisms govern low latency and noise robustness. (1) Silence and voice activity detection: webrtcvad (speech-optimised, fast but inaccurate on music/noise boundaries) and a simple energy threshold (no dependencies, least accurate) were evaluated; silero-VAD is used in production — its deep-learning-based detector correctly gates music clips at the noise boundary, rejecting silent or sub-threshold queries before any feature computation. (2) Loudness normalisation: peak normalisation (avoids clipping but ignores perceived loudness) and EBU R128/LUFS via pyloudnorm (broadcast-standard perceptual accuracy) were considered; RMS normalisation is used as the default for its zero-dependency speed, with EBU R128 as an upgrade path for queries arriving from diverse recording environments. (3) Noise robustness during evaluation and threshold calibration: manual SNR mixing gives full control but is verbose; the audiomentations library is used in production, providing declarative augmentation pipelines (AddGaussianNoise, RoomSimulator, TimeStretch, PitchShift) that simulate the realistic degradations — background noise, reverb, codec artefacts — that 3–10s query clips encounter in the wild. (4) Query routing strategy: always running both pipeline stages maximises accuracy but doubles compute; a parallel race fires both and takes the first result but also doubles compute with no latency floor improvement; hash-first with early exit is used in production — the histogram stage handles the majority of clean queries in under 15ms, and the CNN14 embedding fallback fires only for the minority of noisy clips that fail Stage 1, keeping median end-to-end latency under 50ms on CPU and under 10ms on GPU for the embedding pass.
